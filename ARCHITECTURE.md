# Site architecture: `_data/` vs `assets/`

This site is built with [Jekyll](https://jekyllrb.com/). Two folders often get
confused because both contain things that look like "data" — but they play
**opposite roles**. This note explains the difference so you always know where a
file belongs.

## The one-sentence rule

- **`_data/` = source you edit. Read at build time, never put on the web.**
- **`assets/` = output the site serves. Copied to the live site, fetched by the browser.**

## How they connect (the important bit)

They are not duplicates of each other — they are two ends of a **build pipeline**:

```
_data/  ──►  Jekyll build  ──►  assets/  ──►  browser
(you edit)   (Liquid/plugins)   (published)    (fetch)
```

You edit a `.csv` or `.yml` in `_data/`. At build time Jekyll loads it into the
`site.data` variable, and small Liquid templates in `assets/` re-emit it as JSON
for the live site. Your client-side JavaScript then `fetch()`es that JSON.

**Example — the clue database:**

1. You edit [`_data/clues.csv`](_data/clues.csv) (easy to edit, like a spreadsheet).
2. Jekyll loads it as `site.data.clues`.
3. [`assets/data/clues.json`](assets/data/clues.json) is a 4-line template whose
   whole body is `{{ site.data.clues | jsonify }}` — at build time it becomes the
   JSON copy of your CSV.
4. `assets/js/database.js` and `search.js` `fetch()` that JSON in the browser.

So `_data/clues.csv` and `assets/data/clues.json` hold the *same* data on purpose:
one is the source you edit, the other is the generated copy the site serves. Edit
the CSV; the JSON regenerates on the next build. **Never hand-edit the generated
JSON.**

`assets/data/indicators.json` works the same way, generated from
`_data/indicators/*.yml`.

## What lives in `_data/` (source — never served)

| File / folder | Purpose |
|---|---|
| `clues.csv`, `clues-mycrossword.csv`, `clues-newspapers.csv` | Master clue lists you edit |
| `hidden_clues_291-340.csv` | Working clue/parse data for puzzles 291–340 (different columns from `clues.csv`) |
| `indicators/*.yml` | Cryptic wordplay indicator lists (anagram, container, homophone, …) |
| `navigation.yml` | The header menu — read by `_includes/header.html` as `site.data.navigation` |
| `external_dictionary/` | A **git submodule** ([Boralverse repo](https://github.com/AndrewSlattery/Boralverse)). Deliberately excluded from Jekyll's data loader by `_plugins/skip_external_data.rb`; read directly by the dictionary plugins. |

`_data/` is a **reserved Jekyll folder name** — Jekyll auto-loads everything in it
(`.csv` / `.yml` / `.json`) into `site.data`. Nothing here is ever copied to the
output site, so none of it is downloadable.

## What lives in `assets/` (published — served to the web)

| Folder | Purpose |
|---|---|
| `css/` | Stylesheets |
| `js/` | Client-side scripts (crossword players, search, dictionary, indicators app, …) |
| `ipuz/` | The `.ipuz` puzzle files (hundreds), fetched at runtime by the crossword scripts |
| `data/` | Generated JSON the browser fetches: `clues.json` (from `_data/clues.csv`), `indicators.json` (from `_data/indicators/*.yml`) |
| `boralverse/` | `borlish-dictionary.json` and `boralverse-wiki.json`, generated by the Ruby plugins in `_plugins/` |

Everything in `assets/` is copied verbatim into the built site (`_site/`) and is
downloadable at `https://jackhkeynes.co.uk/assets/...`.

## Why these two folders are **not** merged

They are different Jekyll mechanisms with opposite behaviour, so combining them
would break the site:

- Move `_data/` files into `assets/` → they stop populating `site.data`, so the
  nav menu, clue database, indicators app, search, and Boralverse pages go blank —
  and your raw source CSVs become publicly downloadable.
- Move `assets/` files into `_data/` → the JS, CSS, and puzzles stop being served,
  so the live site loads with no styling, no puzzles, no interactivity.
- `_data` is a *reserved name*; the folder itself can't be renamed without
  rewriting every `site.data.…` reference.

## "Where do I put this new file?" — quick guide

- **Something you'll edit by hand** (a clue list, a word list, menu items)?
  → `_data/`, as `.csv` or `.yml`.
- **Something the browser downloads or loads** (a script, a stylesheet, a puzzle
  file, an image, JSON fetched by JS)? → `assets/`.
- **JSON that's really just your `_data` re-served for the browser?** → a tiny
  Liquid template in `assets/` (front matter + `{{ site.data.X | jsonify }}`),
  following `assets/data/clues.json`. Don't maintain it by hand.
